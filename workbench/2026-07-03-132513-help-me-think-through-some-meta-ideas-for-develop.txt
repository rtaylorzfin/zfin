‚óŹ Good problem to think about. Let me push on it a bit, because I think there
  are two different problems tangled together here, and the best solution
  depends on which one actually hurts more.

  Two separable problems

  Problem A ‚ÄĒ parallel isolation: you can't cleanly have feature-xyz and
  feature-uvw live at the same time.

  Problem B ‚ÄĒ switching cost: deploying/rebuilding to see a change is slow, so
  hopping between branches is expensive.

  Your ideas all attack A. But B is worth naming separately, because if the
  deploy loop were fast, the cost of serial branch-switching drops a lot and
  you'd need less parallel infrastructure. The two also have very different
  price tags ‚ÄĒ A is a real infra project; B is often a config tweak. I'd scope
  both before committing to the container-per-feature build.

  On the container-per-feature idea

  It's the right instinct, but the naive version (N full stacks) will crush you
  on resources, because for ZFIN the expensive tenants are Postgres and Solr,
  not Tomcat. The key move is to split the stack by cost:

  - Shared, always-on infra project: Postgres + Solr. One copy.
  - Per-feature app project: httpd + Tomcat + (compile container), one per
  worktree, cheap to spin up/down.

  That turns your docker compose down feature-xyz; up feature-uvw fallback into
  "just stop the app tier" ‚ÄĒ the heavy stuff never restarts.

  Making it concrete (extends what you already have)

  1. Worktree per feature. This directly solves your "source mounted at one
  path" issue ‚ÄĒ each worktree is a different host path, so each container mounts
  its own path. No conflict.
  2. Parameterize the compose project. You already have
  COMPOSE_PROJECT_NAME=dazed. Make it COMPOSE_PROJECT_NAME=feature-xyz + a port
  offset, one .env per worktree. That gives you N isolated app stacks for free.
  3. Traefik (or Caddy) in front. A single reverse proxy that routes by hostname
  via Docker labels ‚ÄĒ feature-xyz.zfin.test ‚Üí that feature's Tomcat,
  auto-discovered. Wildcard *.zfin.test ‚Üí 127.0.0.1. This is exactly the
  feature-xyz.zfin.org UX you described, and it ties into your existing
  local-dev-hostname revisit note ‚ÄĒ pick .zfin.test so it's unambiguously not
  prod.

  The DB-isolation question (the real hard part)

  Sharing one Postgres across features is fine until a ticket changes schema or
  mutates data ‚ÄĒ then feature-xyz's migration corrupts feature-uvw's world.
  Options, cheapest to most isolated:

  - Share the DB for read-heavy / UI tickets. Simplest.
  - Per-feature DB via CREATE DATABASE ... TEMPLATE ‚ÄĒ fast logical clones inside
  one Postgres instance.
  - ZFS snapshots of the Postgres/Solr data dirs ‚ÄĒ you're on ZFS
  (/opt/zfin/source_roots), so copy-on-write clones of a warmed-up DB + Solr
  index are nearly free and instant. This is probably your best long-term
  primitive: snapshot a "golden" data volume, clone per feature, throw away when
  the ticket merges.

  On running Claude inside each container

  I'd push back gently here. Two models:

  - Claude in-container: full isolation, Claude can run builds/tests in the
  exact env. Cost: N Claude processes, auth/config duplicated, and
  editing-over-mount friction.
  - Claude on host, per worktree (my lean): one Claude terminal tab per worktree
  on the host. It edits files directly (fast), and runs builds via docker 
  compose exec into that feature's container. You get isolation of the
  environment without paying for isolation of the agent. Given your compile
  happens in a container already (per your compile-container login-shell note),
  this composes naturally.

  Don't skip Problem B

  Independent of all the above, attack the deploy loop directly ‚ÄĒ it may be the
  higher-leverage fix:

  - Frontend: you already have gradle dirtycopy refreshing dist/asset-manifest.
  Lean on that + webpack watch so JS/CSS changes don't need a full deploy.
  - Java: class hot-reload (DCEVM/HotSwapAgent, or JRebel) so most Java edits
  skip the Tomcat redeploy cycle. This is the single biggest quality-of-life win
  for a Tomcat app.
  - Measure where the deploy minutes actually go (GWT compile is my suspect)
  before optimizing.

  A couple of other angles

  - Background agents for the long tail. Use Claude's background/scheduled runs
  so a slow build on ticket A proceeds while you drive ticket B ‚ÄĒ parallelism of
  attention, not just infra.
  - Golden data as a first-class artifact. Whatever isolation you pick, having a
  scripted, snapshot-able "known-good Postgres+Solr" state pays off everywhere
  (onboarding, CI, throwaway envs).

  ---
  My recommended path, in order: (1) worktree + parameterized compose project
  name + Traefik routing ‚Üí gets you clean parallel envs; (2) shared infra tier +
  ZFS-snapshot DB clones ‚Üí keeps it affordable; (3) Java hot-reload ‚Üí makes
  each env pleasant to work in.

  Want me to look at your actual docker/docker-compose.yml and .env and sketch
  the parameterized multi-stack + Traefik setup against what's really there?
  That'd turn this from theory into a concrete diff.

‚óŹ "Golden data" = stop treating your dev DB/Solr state as something you restore
  ad hoc, and start treating it as a versioned, tagged artifact you can 
  materialize cheaply on demand ‚ÄĒ same way you treat a build output. Concretely,
  four properties:

  1. It's built by a recipe, not by hand. A scripted pipeline: take a
  prod/unload dump ‚Üí load ‚Üí scrub/anonymize if needed ‚Üí warm it ‚Üí capture. The
  output is the artifact. Anyone can regenerate it; nobody hand-massages a
  database.

  2. It's tagged, and the tag encodes the schema/migration level.
  golden-2026-07-01 at liquibase/migration level N. That's how you know when
  it's stale relative to a branch that adds a migration ‚ÄĒ the branch's
  migrations apply on top of golden, and if they don't, golden needs a rebuild.

  3. It's stored in a portable form. Since you can't count on ZFS, your two
  portable primitives are:

  - Postgres CREATE DATABASE feature_xyz TEMPLATE golden ‚ÄĒ near-instant logical
  clone, works on any filesystem (it's a local file copy, no dump-parse). This
  is your ZFS-snapshot substitute for per-feature DBs within one shared Postgres
  instance. Only caveat: no live connections to the template during the copy.
  - A Docker image with the initialized data baked into the volume layer ‚ÄĒ e.g.
  zfin-db:golden-2026-07-01. Spin-up is instant, it's versioned, and ‚ÄĒ the real
  win ‚ÄĒ it's shareable: push once, the whole team (and CI) pulls the identical
  known-good state. This is the actual "artifact." Size is the tradeoff.

  4. It's refreshed on a cadence ‚ÄĒ a nightly/weekly job rebuilds the golden
  image from fresh-ish data so it doesn't drift.

  How it snaps into your setup

  This reconciles "per-feature DB" + "no ZFS" + "golden data" into one shape:

  - One shared golden Postgres container, started from zfin-db:golden-<tag>.
  - Per feature: CREATE DATABASE feature_xyz TEMPLATE golden_db. Cheap,
  isolated, portable. Your provisioning script (new-feature feature-xyz) does:
  create worktree ‚Üí write .env (loopback alias 127.0.0.x, COMPOSE_PROJECT_NAME,
  port offset) ‚Üí create the templated DB ‚Üí point that feature's Tomcat at it.
  - Migration-touching ticket? Its migrations run against its DB copy; golden is
  untouched.

  Solr matters just as much here

  Your reindex-OOM note is the reason to bake Solr too ‚ÄĒ a 1.35M-doc reindex per
  feature is a non-starter. So the golden artifact is really DB + a matching 
  pre-built Solr core, tagged together. Per feature, either copy the golden core
  dir into a new core, or share the core read-only for tickets that don't touch
  indexed data. Baking the index (not reindexing) is the whole point.

  The payoff is broader than parallel dev: the same tagged golden image makes
  onboarding ("pull and go"), CI, and throwaway experiment envs all reproducible
  ‚ÄĒ including the "Claude inside a container" experiments you want to keep on
  the table, since each one just starts from a known image.

  Want me to look at how your DB and Solr get initialized today (the docker/
  setup and whatever loads the DB) and sketch the golden-image build recipe +
  new-feature provisioning script against your actual files?

‚óŹ This is even cleaner than I expected, because your db and solr images already 
  separate config from data. Today ghcr.io/zfin/zfin-db and zfin-solr ship
  schema/config only ‚ÄĒ the actual data lands at runtime in the pg_data /
  solr_var named volumes via gradle loaddb / loadsolr. "Golden data" is just:
  bake the loaded data into a tagged image variant, and let Docker's overlayfs
  give you copy-on-write per container ‚ÄĒ which is your portable, no-ZFS
  substitute for snapshots.

  The key realization

  You don't need ZFS. When a container starts from an image without a persistent
  volume mounted, its writable layer is copy-on-write over the image layers. So
  a zfin-db-golden:<tag> image with PGDATA baked in gives every feature stack
  its own instant, isolated DB copy ‚ÄĒ created lazily, discarded on down. That's
  exactly the per-feature DB you wanted, and it works on any filesystem.

  Producing the golden images (a repeatable pipeline, ideally a Jenkins job)

  Reuse the load path you already have, then capture the result:

  DB:
  1. Boot an ephemeral zfin-db:${ZFIN_RELEASE} container.
  2. gradle getdb && gradle loaddb && gradle liquibasePreBuild && gradle 
  liquibasePostBuild ‚ÄĒ your existing tasks, to a known migration level.
  3. Stop Postgres cleanly, tar PGDATA.
  4. Build zfin-db-golden:<tag> = base image + COPY that tarball into the data
  path (a tiny Dockerfile, not docker commit ‚ÄĒ reproducible and layer-cached).

  Solr:
  1. Boot zfin-solr:${ZFIN_RELEASE}, gradle getsolr && gradle loadsolr (the ~1‚Äď5
  min replication restore, not the 36-min reindex).
  2. Stop, tar /var/solr.
  3. Build zfin-solr-golden:<tag> = base + COPY.

  You already publish to ghcr.io/zfin and tag by ZFIN_RELEASE, so this slots
  into your existing registry conventions.

  Consuming them (ties into your loopback-alias setup)

  Parameterize the image tags in compose (${ZFIN_DB_IMAGE}, ${ZFIN_SOLR_IMAGE}),
  then a feature .env is just:

  COMPOSE_PROJECT_NAME=feature-xyz
  ZFIN_DB_IMAGE=ghcr.io/zfin/zfin-db-golden:2026-07-01
  ZFIN_SOLR_IMAGE=ghcr.io/zfin/zfin-solr-golden:2026-07-01
  # bind ports/hostname to 127.0.0.x for feature-xyz.zfin.test

  Drop the persistent pg_data/solr_var mounts for throwaway feature stacks (or
  let Compose auto-namespace them as feature-xyz_pg_data). up ‚Üí populated env in
  seconds; down ‚Üí gone. One thing to watch: your db command: pins
  shared_buffers=4GB ‚ÄĒ running several golden DBs at once, add a compose
  override to dial that down per feature.

  Reconciling with a branch's migrations (the important part)

  Tag the golden image with its liquibase level, then on feature up run gradle 
  liquibasePostBuild ‚ÄĒ liquibase skips already-applied changesets and applies
  only your branch's new ones. Golden gets you 99% instantly; the delta tops up
  in seconds. That's the answer to "how does a static snapshot stay useful as
  branches add schema."

  Solr is looser: most branches don't touch schema.xml, so the baked site_index
  is fine as-is. When a branch does change indexed fields, you already have the
  targeted tools ‚ÄĒ gradle solrIndex -PsolrEntities=... for just the affected
  entities, no full 36-min reindex.

  The one real tradeoff

  Image size ‚ÄĒ a full ZFIN DB + 1.3M-doc index baked in is many GB per tag.
  Mitigations: keep golden tags on a retention policy, lean on shared base
  layers (the config images rarely change, so only the data layer differs), and
  rebuild on a cadence (nightly/weekly) so getdb+loaddb never happens on a dev's
  critical path again.

‚óŹ Yes ‚ÄĒ that's exactly right, and it's the whole reason for the per-feature
  loopback IP.

  A Docker published port is a socket endpoint = IP:PORT, not just a port
  number. 127.0.0.2:5432 and 127.0.0.3:5432 are different endpoints, so both can
  bind host port 5432 with no collision. Same for :443, :80, etc. So every
  feature stack can use the identical, memorable port numbers, and the only
  thing that differs is the IP (which the hostname already encodes). That's
  cleaner than port offsets, and it's precisely what the script writes:

  DOCKER_DB_PORT=127.0.0.2:5432        # feature xyz
  DOCKER_HTTPD_HTTPS_PORT=127.0.0.2:443

  DOCKER_HTTPD_HTTPS_PORT=127.0.0.3:443

  them on 127.0.0.1.)

  On Claude-in-container + --dangerously-skip-permissions

  You're right that there's real synergy, and it dovetails with the push
  preference ‚ÄĒ but the container is only a safe boundary if it's built to be
  one. The key caveat:

  Do not YOLO in the compile container. Look at what it mounts:
  - /var/run/docker.sock:/var/run/docker.sock        # = root on the host
  - ${DOCKER_SSH_AUTH_SOCK}:/run/host-services/ssh-auth.sock   # = your push/SSH
  creds
  - ${DOCKER_SOURCE_ROOTS_PATH}:...:rw               # your whole source tree
  - maven_cache / gradle_cache (shared volumes)
  docker.sock alone is a full host escape ‚ÄĒ anything in that container can
  control the host Docker daemon as root. And the SSH agent socket means it can
  push. So --dangerously-skip-permissions there is not sandboxed at all; it's
  arguably more dangerous than on the host.

  What actually makes the container a boundary (a purpose-built claude sidecar,
  not compile):
  - Mount only the one feature worktree :rw. Nothing else.
  - No docker.sock, no ssh-agent socket, no host creds (~/.ssh, ~/.aws,
  gcloud‚Ä¶).
  - Only ANTHROPIC_API_KEY (or your auth) passed in; restricted network egress
  if you want to go further.
  - Talks to the feature's db/solr over the Compose network for testing ‚ÄĒ that's
  the only other thing it needs.

  The elegant part: no credentials in the container makes "human does the push" 
  a hard guarantee, not a soft rule ‚ÄĒ Claude can't push because there's nothing
  to push with. YOLO becomes acceptable because the worst case is a trashed
  disposable worktree (git-tracked, recoverable) rather than a compromised host.
  Defense-in-depth option on top: a PreToolUse hook that blocks git push even
  in skip-permissions mode.

  Tradeoff to weigh: a sandboxed Claude loses your host setup (MCP servers,
  config, tooling), so you'd provision a minimal ~/.claude into the image.

  This is a natural extension of new-feature.sh ‚ÄĒ an optional claude service in
  the feature stack behind a --claude flag / Compose profile. I'd scope it as a
  follow-up rather than building it now, since we've added a lot this session.
  Want me to draft that minimal-mount claude container + the push-blocking hook
  next, or leave it here for now?
