Why patch by section?
3:42 PM

Patch by section because it gives you the best tradeoff between too granular and too bulky.

A generic update-field(field, value) endpoint is easy to build, but weak architecturally: validation becomes scattered, conditional rules are awkward, field renames leak into the API, and related values can be saved in invalid combinations.

A giant PATCH /line-submission/{id} for the whole submission is cleaner than field updates, but it makes autosave heavier and increases merge/conflict risk when multiple sections or child editors evolve independently.

Section-level patches are the nice middle:

PATCH /line-submissions/{id}/overview
PATCH /line-submissions/{id}/background
PATCH /line-submissions/{id}/acceptance-reasons
PATCH /mutations/{id}/general
PATCH /phenotypes/{id}
PATCH /genotyping-assays/{id}
Each endpoint maps to a real user task and a coherent validation boundary. For example, backgroundChangeable and backgroundChangeConcerns can be validated together. Assay type and assay-type-specific fields can be validated together. Phenotype timing unit, start, end, and derived hpf values can be validated together.

So the rule of thumb I’d use is: patch the smallest coherent domain object or form section that has validation rules as a unit. Not one field, not the whole application state.
