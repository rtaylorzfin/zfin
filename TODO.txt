- [ ] space efficiency: preloaded stacks make a FULL COPY of the DB/Solr per feature
      (Docker seeds each <proj>_pg_data/_solr_var by copying the image's baked VOLUME
      data -- NOT copy-on-write). ~26G db + ~10G solr PER feature; 3 stacks ≈ 100G+.
      Explore real per-feature CoW so N clones are near-free:
        - ZFS (the original idea): snapshot a golden loaded pg_data/solr_var dataset,
          `zfs clone` per feature, destroy on teardown. Needs ZFS-backed docker storage
          or bind-mounting ZFS clones as the volumes.
        - btrfs subvolume snapshots, or reflink copies (cp --reflink=auto) on APFS/XFS.
        - Docker doesn't do CoW volume seeding itself, so this likely means managing the
          data dirs outside Docker's named-volume seeding (bind-mount clones) or a
          storage driver / plugin that clones.
      Trade vs current: much less disk + faster "up", at the cost of a fs dependency
      (the very thing the preloaded-image approach was chosen to avoid). Weigh per host.
      HOST OS MATTERS: this is a Linux-host capability. On Linux, ZFS/btrfs on the host +
      bind-mount clones into containers works. On macOS (Docker Desktop) it's effectively
      unavailable -- volumes live inside the LinuxKit VM's disk (not host APFS), you can't
      run ZFS/btrfs in that managed VM, and APFS clonefile doesn't reach Docker volumes;
      bind-mounting a 26G PGDATA over virtiofs is too slow/fsync-fragile for a DB. So CoW
      clones are the win on Linux dev/CI; on macOS, lean on --slim + minimal-table-set.
      (This is exactly why the preloaded full-copy approach was chosen: it works on macOS.)

- [ ] hibernate/archive idle feature stacks to cheap storage (USB/NFS), restore on demand.
      A stack you're not touching still pins ~36G local (26G db + 10G solr). Idea:
      `zfeature archive <stack>` -> stop it, tar its stateful volumes (pg_data, solr_var)
      to an archive dir on USB/NFS, `docker volume rm` locally (frees fast disk; keep the
      tiny worktree/.env/.zenv). `zfeature restore <stack>` -> recreate volumes, extract,
      start. Reuses build-preloaded's exact tar-capture mechanism.
      Speed: minutes each way (compressed ~15G; read from slow media + write 36G local) --
      faster than a full loaddb/reindex, slower than plain `stop`/`start` (which keeps data
      local). So it's the "parking for days/weeks" tier, between stop and full teardown.
      Caveats: archive AT REST on USB/NFS is fine; do NOT run the DB directly off NFS/USB
      (I/O latency + fsync/locking break Postgres). Only pg_data/solr_var need archiving;
      www_data/catalina_base are regenerable via a redeploy.
      KEY INSIGHT: a feature that only changed CODE (not DB data) needs NO db archival at
      all -- its DB is identical to the preloaded image, so "restore" = re-seed from the
      preloaded image (a fresh `up`), and the only state to keep is the git branch. So
      archival is really only for DB-MUTATING features; detecting/limiting to the diverged
      data is the interesting part (ties into minimal-table-set + --slim). Especially
      relevant on macOS where CoW clones aren't an option.

      WHY THE DB/SOLR TRICK DOESN'T TRANSFER (the structural catch): pg_data/solr_var are
      SINGLE-OWNER (only db / only solr), so baking data into that service's image at the
      VOLUME path + letting Docker seed the fresh named volume works cleanly. The deploy
      volumes are SHARED, mounted at the SAME path in many services:
        www_data=/opt/zfin/www_homes/zfin.org      -> compile,db,httpd,certbot,tomcat,tomcatdebug,jenkins (7)
        catalina_base=/opt/zfin/catalina_bases/... -> compile,tomcat,tomcatdebug,jenkins,filebeat (5)
        keystore=/opt/apache/apache-tomcat/conf    -> compile,tomcat,tomcatdebug (3)
        tls_certs=/opt/zfin/tls                     -> compile,httpd (2)
      Docker's implicit image->volume seeding is FIRST-MOUNTER-WINS and copies from THAT
      container's image at the path. For www_data, whichever of db/tomcat/httpd/... starts
      first seeds it -- and most have nothing at that path, so it seeds EMPTY (wrong). So
      you CANNOT rely on image seeding for shared volumes -- populate them EXPLICITLY.

      STRUCTURE -- Option A (recommended; reuses existing pieces):
        - build-preloaded (extended): after a full base deploy, `tar` the deploy-target
          volumes (www_data, catalina_base, keystore, tls_certs) to RESTORE tarballs (same
          capture it does for pg/solr, but NOT baked into a per-service image).
        - new-feature (before `up`): create the per-project volumes and extract each tarball
          into its volume via a throwaway container
          (`docker run --rm -v <proj>_www_data:/dest -v <arch>:/src alpine tar xzf ...`),
          then `up`. Deterministic; no start-order/seeding ambiguity; all sharers see it.
      Option B (more compose-native): a one-shot `app-seed` service (carrying the artifacts
      in an image) that mounts all the shared volumes, `cp -a`'s content in, then exits;
      app services `depends_on: {app-seed: {condition: service_completed_successfully}}`.
      Versioned/self-contained (nice for GoCD), but new image + compose plumbing. Prefer A.

      NOTES: baked deploy = BASE branch at bake time -> feature boots serving main's app,
      then `dirtydeploy` overlays its delta (WARM START, not final). Rebake as main moves.
      Cert is fine to share (generate_base.sh makes CN=zfin.org for every instance anyway).
      Optionally also warm gradle_cache/maven_cache (faster recompile; heavier/churnier ->
