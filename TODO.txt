      are single-arch today — nothing in CI (.github/workflows/{ant,eslint}.yml) builds/pushes them

      pushed), so they need only the builder's native arch — build-preloaded.groovy handles that
      with plain `docker build`.

Bugs found
- [x] FIXED: IP collision: the base zfin_org stack is pinned to 127.0.0.2 (docker/.env
      DOCKER_DB_PORT=127.0.0.2:5432, DOCKER_HTTPD_HTTPS_PORT=127.0.0.2:443), but
      new-feature.groovy's IP allocator starts at 127.0.0.2 too (max octet across
      wt-*/.env, min 1, +1 => .2 for the first feature). So feature #1 (zfh-123)
      binds the SAME host ports as the base -> whichever `up`s its db second fails
      with "port is already allocated". Surfaced trying to validate `zbuild deploy`
      against zfin_org while zfh-123 was up.
      Fix options: (a) allocator also treats the base docker/.env's IP as occupied
      (read DOCKER_DB_PORT/LOOPBACK from it); (b) start feature numbering at .3,
      reserving .1 (loopback default) + .2 (base); (c) move base to 127.0.0.1.
      DONE via (a)++: allocator now gathers taken octets from the base .env, every
      wt-*/.env, AND anything bound on 127.0.0.x right now (docker ps), then scans from a
      configurable start (--ip-base / $ZFIN_FEATURE_IP_BASE, default 2) to the first free
      octet. Validated: default -> .5 (skips base .2 + features), --ip-base 100 -> .100.
- [x] FIXED: mailpit is a global singleton: docker-compose.yml gave it `container_name: mailpit`
      (fixed, not project-prefixed), so only ONE project can run it -- parallel stacks
